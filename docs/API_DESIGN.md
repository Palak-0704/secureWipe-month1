# SecureWipe-AI REST API Design

## Overview
This document defines the backend contract for the secure two-stage wipe flow:

1. In-app live scan and operator selection.
2. Offline re-scan before destructive execution.

All endpoints return JSON except PDF certificate download.

## Two-Stage Device Detection Model

### Stage 1: In-App Detection (Operator Decision Stage)
Triggered when user starts scan in the app.

- Endpoint: `GET /api/devices`
- Optional audit marker: `POST /api/scan/log`
- Purpose:
  - Enumerate currently connected storage devices.
  - Present operator-facing attributes for selection.
  - Record detection confidence per sensitive field.

Expected returned fields include:
- Device identity: `id`, `model`, `serial` (if available)
- Capacity/layout: `size_gb`, `partitions[]`
- Safety attributes: `removable`, `is_system`, `encrypted`, `hpa_dco`
- Confidence model: `detection_confidence`
  - `measured`
  - `inferred`
  - `unknown`

Stage 1 output is not trusted for destructive action by itself. It is used to create a wipe session manifest and operator plan.

During session creation, backend persists a stage-1 detection snapshot chain:
- `target_detection_snapshot` (canonical JSON payload)
- `target_detection_snapshot_sha256`
- `target_detection_snapshot_signature` (when signing seed is configured)

### Stage 2: Offline Detection (Execution Safety Stage)
Triggered after reboot into offline environment, immediately before destructive wipe.

- Session setup and handoff:
  - `POST /api/wipe/session/create`
  - `POST /api/usb/prepare`
- Offline execution:
  - `POST /api/offline/wipe/execute`

Current handoff package generated by `POST /api/usb/prepare` contains:
- `wipe_manifest.json`
- `offline_result_ingest_template.json`
- `README_OFFLINE.txt`
- `RUN_OFFLINE_WIPE.cmd`
- `run_offline_wipe.sh`
- `OFFLINE_RUNTIME_STATUS.txt`
- `offline_runtime.exe` or `offline_runtime` when the binary is available during packaging

`POST /api/usb/prepare` request body includes:
- `session_id` (required)
- `usb_device_id` (required)
- `usb_overwrite_confirmation_text` (optional; required in real provisioning mode when overwrite confirmation policy is enabled)

`POST /api/usb/prepare` response now also returns:
- `provisioning_mode` (`simulation` or `real`)
- `bootable_verified` (boolean flag indicating whether USB bootability checks passed)
- `provision_report_path` (path to JSON provisioning report when real mode is used)

The intended offline operator path is:
- Run the dedicated offline runtime binary against `wipe_manifest.json`.
- The binary re-validates identity, confidence policy, and stage-1/stage-2 snapshot parity.
- The binary emits:
  - `offline_result_ingest.json`
  - `offline_execution_report.json`
- The in-app/backend environment later ingests `offline_result_ingest.json` through `POST /api/offline/result/ingest`.

`offline_result_ingest.json` for verified outcomes must include structured verification evidence:
- `verification_evidence.sample_blocks_checked` (> 0)
- `verification_evidence.sample_blocks_anomalies` (must be 0 when `completion_status=verified`)
- `verification_evidence.checksum_algorithm` (for example `sha256`)
- `verification_evidence.verification_tool`

Packaging behavior:
- `POST /api/usb/prepare` attempts to bundle the offline runtime automatically from the local build output.
- Operators can override the source binary path with `SECUREWIPE_OFFLINE_RUNTIME_BINARY`.
- If the binary is not available, packaging still succeeds but records the gap in `OFFLINE_RUNTIME_STATUS.txt` and the runner scripts fail closed with guidance.

USB provisioning modes:
- Simulation mode (default) writes handoff artifacts and scripts only.
- Real mode executes a host-side provisioning command and stores a provisioning report JSON.
- Real mode controls are configured through environment flags:
  - `SECUREWIPE_USB_PROVISION_MODE=real`
  - `SECUREWIPE_USB_REAL_PROVISION_ENABLED=1`
  - `SECUREWIPE_USB_REAL_BREAKGLASS=1`
  - `SECUREWIPE_USB_REAL_ALLOWLIST=<comma-separated usb ids>`
  - `SECUREWIPE_USB_REQUIRE_OVERWRITE_CONFIRMATION=1`
  - `SECUREWIPE_USB_REAL_PROVISION_COMMAND=<command template>`
  - `SECUREWIPE_USB_REAL_PROVISION_ARGS_JSON=<json array>`
  - `SECUREWIPE_USB_MIN_SIZE_GB=<minimum removable USB size>`
- When overwrite confirmation is required, `usb_overwrite_confirmation_text` must be exactly `ERASE_USB`.
- Real provisioning is fail-closed by default and requires both break-glass enablement and explicit USB device allowlisting.
- Backward compatibility: legacy aliases `SECUREWIPE_USB_PROVISION_COMMAND` and `SECUREWIPE_USB_PROVISION_ARGS_JSON` are still accepted.

Before calling the wipe engine, backend performs a fresh `detect_devices()` scan again and enforces:
- Target exists in runtime scan.
- Target identity matches manifest (id/model/size and serial when present).
- Stage-2 runtime snapshot hash matches the recorded stage-1 snapshot hash.
- If present, stage-1 snapshot signature verifies and matches the recorded hash.
- Protected system disk policy.
- Strict targeting policy (removable or allowlisted).
- Detection confidence policy for critical fields.

If any guard fails, wipe is blocked with typed API errors and audit history entries.

## Safe Handoff to Wipe Engine

Destructive wipe execution is allowed only after API-level and engine-level checks.

### API-Level Guards
Implemented in offline handlers before engine invocation:
- Confirmation text must be `ERASE`.
- Session phase transition must be valid.
- Runtime device identity must match session manifest.
- Detection confidence must be sufficient unless explicit override is configured.

### Engine-Level Guards
Enforced in wipe engine regardless of API request:
- Runtime mode must be offline.
- Destructive feature/env flags must be enabled.
- Protected/system target blocks are re-checked.

### Offline Runtime Binary
Current operator binary: `cargo run -p securewipe-cli --bin offline_runtime -- --manifest <path> --confirmation-text ERASE --output-dir <dir>`

This binary is intentionally separate from the REST API so the destructive runtime can operate in an offline environment while still using the same manifest, result schema, and chain-of-custody checks.

This is a fail-closed chain: both layers must allow execution.

## Error and Policy Notes

- Validation failures use typed codes (for example `*_required`, `*_mismatch`, `*_invalid`).
- USB preparation can return `usb_overwrite_confirmation_required` when overwrite confirmation policy is active and confirmation text is missing/invalid.
- USB preparation can return `usb_device_not_suitable` when target media is not removable or below minimum size policy.
- USB preparation can return `usb_real_provisioning_not_enabled` when real provisioning command execution is not explicitly enabled.
- USB preparation can return `usb_real_provision_command_missing` when real mode is enabled without a configured command template.
- USB preparation can return `usb_real_breakglass_required` when real mode is requested without `SECUREWIPE_USB_REAL_BREAKGLASS=1`.
- USB preparation can return `usb_real_allowlist_required` when real mode is requested without `SECUREWIPE_USB_REAL_ALLOWLIST`.
- USB preparation can return `usb_device_not_allowlisted_for_real_provision` when the selected USB ID is not explicitly allowlisted.
- Verified ingest payloads can return `verification_evidence_required`, `verification_evidence_invalid`, or `verification_evidence_inconsistent` when evidence is missing or contradictory.
- High-risk operations can return `detection_confidence_insufficient` when critical confidence is unknown.
- Offline execution can return `offline_stage1_snapshot_tampered`, `offline_stage1_snapshot_signature_invalid`, or `offline_stage2_detection_mismatch` when chain-of-custody checks fail.
- Unknown-confidence override is intentionally explicit via environment flags and should only be used in controlled emergency workflows.
